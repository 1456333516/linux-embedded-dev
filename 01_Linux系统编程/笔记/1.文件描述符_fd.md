# 1.文件描述符

## 1.1 核心概念

### 1.1.1 什么是文件描述符（fd）？

在 Linux 内核中，每当进程打开一个文件或设备时，内核就会向进程返回一个小的非负整数，这就是 fd。你可以把它看作是内核给这个设备发的一个“取件码”。后续要对这个设备读数据、写数据，全凭这个“取件码”。



### 1.1.2 默认的三个“取件码” (0, 1, 2)

当你在 Linux 下启动一个程序时，系统会自动帮你打开三个 fd：

- **0 (STDIN_FILENO)**：标准输入（通常是键盘）。
- **1 (STDOUT_FILENO)**：标准输出（通常是终端屏幕）。你平时用的 `printf` 或 `std::cout`，底层就是把数据写到了 fd 1。
- **2 (STDERR_FILENO)**：标准错误（也是屏幕，但专门用来输出报错信息）。



### 1.1.3 问题汇总

#### ① 为什么 Camera 也是文件？

当你在类似 Jetson 这样的开发板上插上一个标准 USB 摄像头时，Linux 内核的 UVC 驱动识别到硬件后，会在 `/dev` 目录下生成一个设备节点，通常是 `/dev/video0`。 对于你的程序来说，你不需要懂底层的 USB 协议，你只需要像打开一个普通文本文件一样去 `open("/dev/video0")`，内核就会返回一个 fd。你对这个 fd 进行的后续操作，内核会自动帮你转换为对摄像头硬件的操作。





#### ②在 Linux 的单线程或多线程环境下，同一个进程内多次打开不同的文件或设备，系统分配的文件描述符（fd）是否能保证唯一且绝对不重复？

在同一个进程内（无论单线程还是多线程），同时处于打开状态的多个 fd，其值是绝对“不重复”的

**核心铁律：fd 是属于“进程”的，而不是“线程”的**

在 Linux 内核中，**每一个“进程”（Process）都有自己独立的一张“文件描述符表”**。你可以把这张表想象成一面墙的快递柜，编号从 0 到 1024（默认上限）。 多线程（Threads）是寄生在进程里的。一个进程里的所有线程，**共享这同一张“文件描述符表”**。



**单线程情况：最低可用原则 (Lowest Available)**

当你在单线程里连续调用 `open()` 时，Linux 内核分配 fd 的规则永远是：**寻找当前这面墙上，编号最小且空闲的那个柜子。**

- 系统启动，0、1、2 已经被标准输入/输出/错误占用了。
- 你第一次 `open()` 摄像头 `/dev/video0`，内核巡视一圈，发现最小的空位是 3。于是返回 `fd = 3`。
- 你第二次 `open()` 某个日志文件 `log.txt`，内核发现 3 已经被占用了，最小空位是 4。于是返回 `fd = 4`。
- **关键点（重复利用）**：如果你此时执行了 `close(3)`（关掉了摄像头），3 号柜子就空出来了。你再去 `open()` 另一个文件时，内核会再次把 `fd = 3` 分配给你！

**总结**：只要你没 `close()`，一直 `open()` 获取的 fd 绝对是不重复的（3, 4, 5, 6...）。



**多线程情况：内核级的绝对互斥**

既然所有线程共享同一张表，那假如 **线程 A 和线程 B 在同一微秒** 同时调用了 `open()` 呢？会不会都拿到 `fd = 3`？

**绝对不会。** Linux 内核在处理 `open()` 这个系统调用时，是带有“锁”的（原子操作）。内核会排队处理这些请求：

- 哪怕 A 和 B 同时发起，内核也会瞬间决定一个先后顺序。
- 比如先处理 A，给 A 分配了 3 号；紧接着处理 B，给 B 分配 4 号。 所以，在多线程环境下，你获取到的 fd 依然是**安全、唯一、不重复的**。



 **真正会“重复”的情况：多进程 (Multi-Process)**

如果你启动了两个**完全不同的程序**（比如你运行了你写的 `Camera.exe`，同时又运行了一个 `Control.exe`），它们是两个独立的进程。

- `Camera.exe` 进程有一张自己的表，它 `open` 获取到的第一个 fd 是 **3**。
- `Control.exe` 进程也有一张自己的表，它 `open` 获取到的第一个 fd 也是 **3**。 这两个 `3` 互不干扰，代表着各自进程里打开的不同文件。就像 A 小区的 3 号快递柜和 B 小区的 3 号快递柜一样。

## 1.2 四大核心系统调用 (API)

在 C/C++ 中，这四个函数包含在 `<unistd.h>` 和 `<fcntl.h>` 头文件中。

### 1.2.1 `open()` —— 获取 fd

在 Linux “一切皆文件”的哲学下，无论是普通的文本文件（如 `.log`、`.json`），还是底层的硬件设备（如 USB 摄像头 `/dev/video0`、串口 `/dev/ttyUSB0`），我们在代码中与它们建立连接的第一步，都是调用 `open()` 函数。

#### ① 语法

```c
#include <fcntl.h>

// 基础用法：打开已存在的文件或设备
int fd = open(const char *pathname, int flags);

// 进阶用法：如果文件不存在需要创建时，必须带上第三个参数设置权限
int fd = open(const char *pathname, int flags, mode_t mode);

// 以读写模式打开摄像头。若没数据则阻塞等待。
int camera_fd = open("/dev/video0", O_RDWR);
if (camera_fd == -1) {
    perror("无法打开摄像头设备");
}

// 读写模式 + 非阻塞。没数据时立刻返回，不卡死程序。
int sensor_fd = open("/dev/ttyTHS1", O_RDWR | O_NONBLOCK);
if (sensor_fd == -1) {
    perror("无法打开传感器串口");
}

// 只写模式 + 若无则创建 + 追加写入。设置权限为 0644 (所有者读写，其他人只读)
int log_fd = open("system_error.log", O_WRONLY | O_CREAT | O_APPEND, 0644);
if (log_fd == -1) {
    perror("日志文件创建/打开失败");
}

// 只写模式 + 若无则创建 + 清空旧内容。
int temp_fd = open("temp_buffer.bin", O_WRONLY | O_CREAT | O_TRUNC, 0644);
```

**返回值解析**

- **成功**：返回一个**非负整数**，即**文件描述符（fd）**。内核会分配当前进程中最小且未被占用的编号（通常大于 2，因为 0、1、2 已被标准输入输出占用）。
- **失败**：返回 **`-1`**。此时可以通过查询全局变量 `errno`（或使用 `strerror(errno)` / `perror()`）来获取具体的失败原因（如权限不足、设备不存在等）。

#### ② 基础打开模式（三选一，必须指定）

在调用 `open()` 时，`flags` 参数**必须且只能**包含以下三个基础模式中的一个，用于向内核声明你对该文件/设备的基本操作权限。

- **`O_RDONLY` (Read Only)**：**只读模式**。
  - *适用场景*：读取配置文件（如 `.yaml`）、读取固定的参数表，防止程序意外篡改文件内容。
- **`O_WRONLY` (Write Only)**：**只写模式**。
  - *适用场景*：向屏幕输出特定信息、单纯记录系统运行日志。
- **`O_RDWR` (Read & Write)**：**读写模式**。
  - *适用场景*：**绝大多数硬件设备的标准打开方式**。例如操作摄像头 `/dev/video0` 时，你不仅需要读取（拉取图像数据），还需要写入（下发分辨率、曝光率等控制指令）。

#### ③ 附加打开模式（可选项，可叠加）

除了基础模式，你可以使用 C/C++ 的**位或运算符 `|`**，为 `open()` 叠加各种附加功能。

**文件生命周期管理**

- **`O_CREAT` (Create)**：如果目标文件不存在，则自动创建它。
  - *注意*：使用此参数时，**必须**传入 `open` 的第三个参数（如 `0644`）来指定新文件的系统读写权限。
- **`O_TRUNC` (Truncate)**：如果文件已存在且有内容，打开时将其“截断”为 0 字节（即清空内容）。
  - *适用场景*：每次程序启动时，希望覆盖旧的日志，记录全新的日志。
- **`O_APPEND` (Append)**：追加模式。写入的数据会自动添加在文件的最末尾，不会覆盖原有内容。
  - *适用场景*：长期运行的服务，需要不断向同一个日志文件中追加报错记录。

**硬件控制核心（划重点）**

- **`O_NONBLOCK` (Non-Blocking)**：**非阻塞模式**。
  - *底层逻辑*：默认情况下，Linux I/O 是阻塞的。如果你去 `read()` 一个没有新数据的摄像头或串口，程序就会卡死在这一行死等。加上 `O_NONBLOCK` 后，如果没有数据，`read()` 会立刻返回 `-1`，程序可以继续执行下一行代码。
  - *适用场景*：在自动驾驶、机器人等要求高实时性的系统中，主线程绝不能因为等待某个传感器的数据而卡死，非阻塞模式是必选项。

### 1.2.2 错误处理与 `errno`

一旦返回 `-1`，你怎么知道是设备被占用了，还是权限不够？这时需要查全局变量 `errno`。

```c
#include <errno.h>
#include <string.h>

if (fd == -1) {
    // strerror 会把错误码翻译成人类可读的字符串，比如 "Permission denied"
    printf("打开摄像头失败，错误原因: %s\n", strerror(errno)); 
}
```

`errno`**不需要你自己定义**！要你写了 `#include <errno.h>`（或者 C++ 里的 `#include <cerrno>`），系统就已经帮你把它定义好了。你直接拿来用就行。

现在每个线程都有一个**专属于自己的 `errno`（线程局部存储，Thread-Local Storage）**。当你在代码里写下 `printf("%s", strerror(errno))` 时，编译器会自动把它展开成一个函数调用。系统会精准地去取**当前这个线程**刚才发生的错误码，绝对不会和别的线程串号！

既然不需要自己定义，用的时候就有一条铁律：**只有当系统调用（如 `open`、`read`）明确返回了代表失败的值（通常是 `-1`）之后，你才能去读 `errno`！** 要不然那可能是上一行、甚至十分钟前别的某个函数失败留下的“案底”。

### 1.2.3 `read()` 与 `write()` —— 数据交互

#### ① 语法规则

这两个函数的原型长这样（包含在 `<unistd.h>` 中）：

```c
ssize_t write(int fd, const void *buf, size_t count);
ssize_t read(int fd, void *buf, size_t count);

// 假设你想往文件里写数据
char buffer[] = "Hello System!";
ssize_t bytes_written = write(fd, buffer, sizeof(buffer));

// 从设备/文件读数据
char read_buf[100];
ssize_t bytes_read = read(fd, read_buf, sizeof(read_buf));
```

-  int fd:“取件码/文件描述符”

- void *buf(内存地址):

  - `write` 时：这是你的发货车厢，里面装满了你要发出去的数据（比如 `"Hello System!"`）。

  - `read` 时：这是你的空车厢，你开过去让系统把读到的数据塞进这个内存地址里（比如那个 `char read_buf[100]`）。
  - 注：`void *` 意味着它可以接受任何类型的数据首地址（char数组、int变量、甚至是一个结构体）。

- `size_t count`（期望处理的字节数）

  - `write` 时：你告诉系统“我这车货有 13 个字节，请帮我全写进去”。
  - `read` 时：你告诉系统“我这个空车厢最多只能装 100 个字节，再多就溢出（内存越界）了，最多给我塞 100 个”。

- 返回值 `ssize_t`：极其重要的“实际交接单”

  - **`> 0`（正数）**：**实际**成功读/写的字节数。实际数量可能小于你要求的数量！
  - **`0`**：
    - 对于 `read`：表示**“读到了文件末尾（EOF）”**。就好像仓库空了，啥也没拿到。
    - 对于 `write`：表示什么也没写进去。
  - **`-1`**：报错了！这时候 `errno` 就派上用场了，赶紧用 `strerror(errno)` 查明原因。

**以上最基础的读写函数。但是在操作摄像头通常会用更高级的 `ioctl`（比如 V4L2 框架）或者内存映射（mmap），**

### 1.2.4 `close()` —— 释放资源

用完设备必须关掉，否则会导致“文件描述符泄漏”（系统默认一个进程最多只能打开 1024 个 fd）。

```c
close(fd);
```

### 1.3 代码

#### 1.3.1 fd_test.cpp

```cpp
#include <iostream>
#include <fcntl.h>   // 包含 open
#include <unistd.h>  // 包含 read, write, close
#include <cerrno>    // 包含 errno
#include <cstring>   // 包含 strerror

int main() {
    // 1. 尝试打开一个不存在的设备，测试报错机制
    int bad_fd = open("/dev/video99", O_RDWR);
    if (bad_fd == -1) {
        std::cerr << "预期的失败：无法打开 /dev/video99, 错误原因: " << strerror(errno) << std::endl;
    }

    // 2. 创建并打开一个普通文件测试
    int fd = open("test_camera_log.txt", O_CREAT | O_WRONLY | O_TRUNC, 0644);//数字前面加 0，代表这是一个“八进制（Octal）”数。
    if (fd == -1) {
        std::cerr << "创建文件失败!" << std::endl;
        return -1;
    }

    std::cout << "成功打开文件，分配到的文件描述符 fd = " << fd << std::endl;

    // 3. 写入数据
    const char* msg = "模拟摄像头初始化完成...\n";
    write(fd, msg, strlen(msg));
    
    // 4. 标准输出也是 fd! 我们向 1 号 fd 写点东西
    const char* screen_msg = "这是直接往 fd=1 (屏幕) 写入的文本。\n";
    write(1, screen_msg, strlen(screen_msg));

    // 5. 关闭资源
    close(fd);
    std::cout << "资源已释放。" << std::endl;

    return 0;
}
// 编译命令: g++ test_fd.cpp -o test_fd
// 运行命令: ./test_fd
```
