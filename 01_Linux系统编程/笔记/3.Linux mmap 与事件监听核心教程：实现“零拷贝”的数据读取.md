# Linux `mmap` 与事件监听核心教程：实现“零拷贝”的数据读取

在处理摄像头视频流、音频流或高速外设时，传统的 `read()` 和 `write()` 函数往往会成为性能瓶颈。为了解决这个问题，Linux 提供了 `mmap`（内存映射）机制。

## 1. 什么是 `mmap`？

**核心定义：** `mmap` 能够将一个文件或设备内存（如摄像头的硬件缓冲区）直接映射到你当前进程的虚拟地址空间中。

**类比 STM32：** 如果你写过单片机代码，一定熟悉直接定义一个指针指向某个外设寄存器的绝对物理地址，然后直接读写这个指针来控制硬件或读取数据。在 Linux 中，由于有 MMU（内存管理单元）和虚拟内存的隔离，应用层不能直接访问物理地址。`mmap` 的作用就是搭建一座桥梁，让系统给你分配一个虚拟地址指针，这个指针在底层直接指向了设备的物理内存。虽然它比 STM32 的直接寻址更通用、更复杂（由操作系统管理），但**本质的爽感是一样的：直接通过指针操作底层内存。**

## 2. 为什么要用 `mmap`？（V4L2 场景中的杀手锏）

使用普通的 `read()` 读取摄像头数据时，数据流向是： `摄像头硬件 -> 内核空间缓冲区 -> (CPU 执行数据拷贝) -> 用户空间缓冲区`

使用 `mmap` 后，数据流向变成了： `摄像头硬件 -> 物理内存` 

你的程序和底层驱动**共享**这块物理内存。摄像头把图像数据写入这里，你的程序直接从这里读取。这就实现了**“零拷贝”（Zero-copy）**，极大地节省了 CPU 资源和时间，这对于高帧率的视频流至关重要。

在 Jetson 这种板卡上，CPU 和 GPU **物理上共享同一块内存芯片（LPDDR RAM）**。

在这种架构下，GPU 共享 V4L2 缓冲区的潜力被发挥到了极致：

- **物理层共享**：当 USB 摄像头或 MIPI 摄像头通过底层 DMA（直接内存访问）把图像数据写入系统物理内存时，这块内存不仅可以通过 `mmap` 映射给 CPU 看到，**GPU 也能直接寻址这块物理内存**。
- **如何实现**：在 Linux 系统中，现在更推荐使用 **`dma-buf`**（DMA 缓冲区共享机制）而不是单纯的 `mmap`。通过 V4L2 导出 `dma-buf` 的文件描述符（fd），然后将这个 fd 直接传递给 CUDA（通过 EGLStream 或直接映射）或 TensorRT。
- **结果**：摄像头写入数据 -> GPU 上的 CUDA 核函数或 YOLO 模型直接在原地（In-place）读取并处理这帧图像。CPU 在整个过程中连一个字节都不需要碰，只负责发送控制指令，这是完美的零拷贝。

## 3. 核心 API

### 3.1 建立映射：`mmap`

```c
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);

// 1. 声明一个指针，用来接住映射后的首地址
void *mapped_ptr;

// 2. 执行映射
mapped_ptr = mmap(
    NULL,                             // 让系统分配地址
    buf.m.planes[0].length,           // 映射多大
    PROT_READ | PROT_WRITE,           // 可读可写
    MAP_SHARED,                       // 必须共享
    fd,                               // 摄像头 fd
    buf.m.planes[0].m.mem_offset      // 硬件给的物理偏移量
);

// 3. 习惯性防错检查
if (mapped_ptr == MAP_FAILED) {
    perror("mmap 失败了，检查一下 offset 或 fd 对不对");
}
```

- **`addr`**：通常填 `NULL`，意思是让 Linux 内核自己帮你找一块合适的虚拟地址来映射。
- **`length`**：你要映射的内存大小（也就是你的缓冲区 `size`）。
- **`prot`**：权限设置。通常设为 `PROT_READ | PROT_WRITE`，表示这块映射的内存既能读也能写。
- **`flags`**：**非常关键！** 通常设为 `MAP_SHARED`。这表示对此区域的修改是共享的，也就是说，如果底层驱动更新了这块内存（比如写入了新一帧图像），你的进程能立刻看到；反之亦然。
- **`fd`**：文件描述符。比如你通过 `open("/dev/video0", ...)` 打开摄像头得到的那个数字。
- **`offset`**：偏移量。在 V4L2 中，驱动会通过其他 API 告诉你这个偏移量是多少，直接填进去即可。

**返回值：** 成功时，返回一个指向映射区域的指针（你可以直接用这个指针取数据了！）；失败时返回 `MAP_FAILED`。

### 3.2 解除映射：`munmap`

```c
int munmap(void *addr, size_t length);
```

用完之后必须释放资源。传入你刚才得到的指针 `ptr` 和大小 `size` 即可。



## 4. 怎么知道设备有数据了？（事件监听）

指针虽然拿到了，但你不能写个死循环去不停地检查内存里有没有新图像，那样会把 CPU 占满。你需要一种机制让程序“睡眠等待”，直到外设（如摄像头）说：“我准备好数据了，你来读吧！”。

这就需要用到文件描述符的监听机制。

### 4.1 方式一：经典老将 `select`

- **`FD_SET(fd, &set)`**：把你关心的那个摄像头 `fd` 加入到一个“监听集合”（set）中。
- **`select(...)`**：调用此函数，程序会阻塞（暂停运行），直到集合中的某个 `fd` 发生了读写事件或超时。
- **`FD_ISSET(fd, &set)`**：`select` 返回后，用这个宏来检查到底是不是你的摄像头 `fd` 来数据了。



### 4.2 方式二：更现代的替代品 `poll`

`poll` 的逻辑和 `select` 类似，但在处理多个文件描述符时性能更好，且参数结构更清晰，没有 `select` 那种最大监听数量（通常是 1024）的硬性限制。

```C
struct pollfd fds[1];
fds[0].fd = fd;           // 你要监听的设备
fds[0].events = POLLIN;   // 你关心的是“有数据可读” (IN)

int ret = poll(fds, 1, timeout_ms);
```

**判断返回值（`select` 和 `poll` 通用逻辑）：**

- **`> 0`**：正数表示有事件发生！此时你可以安全地去读取那个 `mmap` 出来的指针了。
- **`== 0`**：超时了。在指定的 `timeout_ms` 时间内，设备没有任何动静。
- **`< 0`**：出错了（通常是 `-1`），需要检查 `errno` 看看发生了什么异常。

```c
// 假设你想持续采集视频
bool is_running = true;

while (is_running) {
    // 1. 让程序睡觉，直到摄像头说：“我弄好一帧了！”
    int ret = poll(fds, 1, 3000); // 3000ms 超时

    if (ret == 0) {
        std::cout << "等了3秒还没图，超时了！" << std::endl;
        continue;
    } else if (ret < 0) {
        perror("poll 报错，可能被信号打断");
        break;
    }

    // 2. 确实有事件发生了，进一步确认是不是“有数据可读 (POLLIN)”
    if (fds[0].revents & POLLIN) {
        
        // --- 下面是拿到数据后的经典三步曲 ---

        // 第一步：出队 (Dequeue Buffer)
        // 告诉驱动：“把那个填满新鲜图像的缓冲区交给我”
        // ioctl(fd, VIDIOC_DQBUF, &buf); 

        // 第二步：使用 mmap 指针处理数据 (零拷贝的爽点就在这)
        // 你直接读 mapped_ptr 指向的内存即可！
        // 比如：扔给 OpenCV 处理，或者送给 RK3588 的 NPU 跑目标检测
        // process_image(mapped_ptr, buf.length); 

        // 第三步：入队 (Queue Buffer)
        // 告诉驱动：“这块内存我用完了，你拿回去继续装下一帧图像吧”
        // ioctl(fd, VIDIOC_QBUF, &buf); 
    }
}
```



## 5. 总结流程

“整个 V4L2 摄像头读取的流程，核心就是通过**内存映射**来实现**零拷贝**。

首先是**准备阶段**：我先 `open` 打开设备节点拿到 fd，向驱动申请好缓冲区。最关键的是，利用驱动返回的物理偏移量，调用 `mmap` 把内核的物理内存直接映射到咱们用户空间，拿到一个可以直接操作的指针。

然后开启视频流，进入**循环阶段**：在这个死循环里，我会先用 `poll` 函数阻塞等待，这样设备没出图的时候 CPU 就可以挂起休息。一旦 `poll` 监听到数据就绪，我就执行‘出队 (DQBUF)’操作把这块填满图像的缓冲区拿出来。因为之前做了 `mmap`，我直接用那个指针就能把图像喂给算法处理，全程不需要 CPU 搬运数据。处理完之后，再把这块内存‘入队 (QBUF)’还给底层驱动，让它去装下一帧图像。

最后是不采了的**收尾阶段**：退出循环后，调用 `munmap` 解除映射，再 `close` 关掉设备文件就可以了。”

总结来说，就是用 `mmap` 解决数据拷贝的性能痛点，用 `poll` 解决死循环查询的 CPU 占用痛点，再配合内核缓冲区的‘出队入队’机制让数据流转起来。
