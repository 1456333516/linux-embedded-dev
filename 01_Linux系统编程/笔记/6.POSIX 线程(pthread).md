# POSIX 线程(pthread)

## 1. POSIX 线程是什么？

### 1.1 什么是 POSIX？（一个标准）

**POSIX** 的全称是 **P**ortable **O**perating **S**ystem **I**nterface（可移植操作系统接口）。

- 它不是一个软件，而是一套由 IEEE（电气和电子工程师协会）制定的**国际标准**。
- **为什么需要它？** 早期的 Unix 系统五花八门，开发者在一个系统上写的代码，换到另一个系统就跑不起来。POSIX 就是为了解决这个问题诞生的“统一普通话”。
- 只要操作系统（比如 Linux、macOS 等 Unix-like 系统）遵循 POSIX 标准，开发者按照这个标准写的 C/C++ 程序就可以在这些系统之间无缝移植。



### 1.2 什么是线程 (Thread)？（最小执行单元）

在操作系统中，**进程 (Process)** 是分配资源的基本单位，而**线程 (Thread)** 是 CPU 调度的最小单位。

- **打个比方**：如果把你的整个 C++ 程序看作一家**工厂**（进程），工厂里有厂房、机器、物料（内存、文件描述符等资源）。那么线程就是工厂里的**工人**。

- 一个工厂至少有一个工人（主线程）。如果为了提高效率，你可以招募更多工人（创建子线程）。这些工人共享工厂里的所有机器和物料（共享内存空间），但每个工人有自己的工作进度表（线程栈、寄存器）。

### 1.3 所以，什么是 pthread？（Linux C 原生多线程库）

**pthread** 就是 **POSIX Threads** 的缩写。 简单来说，它是 POSIX 标准中专门用来**规定如何操作线程**的那部分接口。

- **在 Linux 世界里的地位**：在 Linux 系统下进行 C/C++ 开发时，`pthread` 就是操作系统提供给你的**最原生、最底层的多线程武器**。你在代码里 include 的 `<pthread.h>`，就是这套标准的具体实现。

- **为什么它在高性能开发中无处不在？** 在处理复杂的硬件交互时（比如在 Linux 板卡上一边用摄像头采集图像，一边进行视频硬件编码，同时还要通过串口与其他设备通信），单线程顺序执行会导致严重的卡顿。`pthread` 允许你让这些任务并行跑在 CPU 的不同核心上，并且因为它是 C 语言原生的，开销极小，执行效率极高。

**总结**：**pthread** 就是一套**在 Linux 等类 Unix 系统上，用 C 语言编写高并发、多任务程序的官方标准工具箱**。后来 C++11 引入的 `std::thread`，在 Linux 环境下，其底层本质上依然是对 `pthread` 的一层面向对象的封装。



## 2. 编译前提

在使用 `pthread` 库时，必须在编译时链接线程库。

- **gcc/g++ 命令行**：编译时加上 `-pthread` 参数。 `g++ main.cpp -o main -pthread`

- **CMakeLists.txt**：

  ```cmake
  find_package(Threads REQUIRED)
  target_link_libraries(your_target_name Threads::Threads)
  ```



## 3. 线程的创建与等待

在 Linux 中，主程序运行本身就是一个主线程。我们需要使用 `pthread_create` 开启新的分支，并用 `pthread_join` 回收资源。

### 3.1 创建线程：`pthread_create`



```C
int pthread_create(pthread_t *thread, const pthread_attr_t *attr,
                   void *(*start_routine) (void *), void *arg);
```

- **`thread`**: 指向 `pthread_t` 变量的指针，用于存放新线程的 ID。
- **`attr`**: 线程属性（如栈大小、优先级），通常填 `NULL` 使用默认属性。
- **`start_routine`**: 线程要执行的**函数指针**。注意它的签名必须是 `void* func(void* arg)`。
- **`arg`**: 传递给线程函数的参数。如果是多个参数，通常打包成一个结构体传进去。

### 3.2 等待线程：`pthread_join`

```C
int pthread_join(pthread_t thread, void **retval);
```

- **作用**：阻塞当前线程（通常是主线程），直到指定的 `thread` 结束。这非常重要，如果不 `join`（且没有设置为分离状态），线程结束后会变成“僵尸线程”，导致内存泄漏。
- **`retval`**: 获取线程函数的返回值，不需要的话填 `NULL`。

### 3.3 **代码示例：基础线程创建**

```c++
#include <iostream>
#include <pthread.h>
#include <unistd.h>// Unix 标准头文件，包含 sleep, fork, read, write 等底层系统调用

// 线程执行的函数
void* my_thread_func(void* arg) {
    int* num = (int*)arg;
    std::cout << "子线程运行中，传入的参数是: " << *num << std::endl;
    sleep(1); // 模拟耗时操作
    std::cout << "子线程即将退出" << std::endl;
    return NULL;
}

int main() {
    pthread_t tid;
    int pass_value = 42;

    // 1. 创建线程
    if (pthread_create(&tid, NULL, my_thread_func, &pass_value) != 0) {
        std::cerr << "线程创建失败!" << std::endl;
        return -1;
    }

    std::cout << "主线程: 等待子线程结束..." << std::endl;
    
    // 2. 阻塞等待子线程回收
    pthread_join(tid, NULL);
    std::cout << "主线程: 子线程已回收，程序结束。" << std::endl;

    return 0;
}
```

## 4. 守护临界区：互斥锁（Mutex）

**互斥锁就相当于单片机（如 STM32）中的“关中断”**。 在单片机裸机中，为了防止中断打断主循环对全局变量的修改，我们会关闭中断。而在多核 Linux 系统中，多个线程可能同时跑在不同的 CPU 核心上，关中断无效，必须用互斥锁来保证**同一时刻只有一个线程能访问共享资源（临界区）**。

### 4.1 核心函数

- **初始化**: `pthread_mutex_init(&mutex, NULL);` (或者静态初始化 `pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;`)
- **加锁**: `pthread_mutex_lock(&mutex);` （如果锁已经被别人拿走，当前线程会在这里**阻塞挂起**，交出 CPU 使用权，直到锁被释放）
- **解锁**: `pthread_mutex_unlock(&mutex);`
- **销毁**: `pthread_mutex_destroy(&mutex);`

### **4.2 代码示例：保护共享数据**

```c++
#include <iostream>
#include <pthread.h>

int shared_counter = 0; // 全局共享资源
pthread_mutex_t counter_mutex = PTHREAD_MUTEX_INITIALIZER; // 静态初始化锁

void* increment_func(void* arg) {
    for (int i = 0; i < 100000; ++i) {
        // 进入临界区前加锁
        pthread_mutex_lock(&counter_mutex);
        
        shared_counter++; // 临界区：只有拿到锁的线程才能执行到这里
        
        // 离开临界区后解锁
        pthread_mutex_unlock(&counter_mutex);
    }
    return NULL;
}

int main() {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, increment_func, NULL);
    pthread_create(&t2, NULL, increment_func, NULL);

    pthread_join(t1, NULL);
    pthread_join(t2, NULL);

    // 如果没有互斥锁，结果大概率不是 200000
    std::cout << "最终计数器值: " << shared_counter << std::endl; 
    return 0;
}
```

## 5. 线程间的信号枪：条件变量（Condition Variable）

互斥锁解决了“冲突”问题，但解决不了“协同”问题。 假设你的项目中有一个**编码线程（Encoder）\**和一个\**摄像头采集线程（Camera）**。编码线程需要等摄像头采集到新一帧图像才能开始工作。

- **错误做法（轮询）**：编码线程写个 `while(true) { if (has_new_frame) break; }`。这叫“忙等”，会把 CPU 占用率直接干到 100%。
- **正确做法（条件变量）**：编码线程去睡觉（阻塞挂起），摄像头采集完一帧后，开一枪（发送信号），把编码线程叫醒。

条件变量**必须**配合互斥锁一起使用。

### 5.1 核心函数

- **初始化**: `pthread_cond_init(&cond, NULL);`
- **等待信号 (消费者)**: `pthread_cond_wait(&cond, &mutex);`
  - *重点机制：调用此函数时，线程会自动**解开 `mutex` 锁**并进入睡眠。当收到信号被唤醒时，它会自动**重新获取 `mutex` 锁**。*
- **发送信号 (生产者)**: `pthread_cond_signal(&cond);` （唤醒一个等待的线程） 或 `pthread_cond_broadcast(&cond);` （唤醒所有等待的线程）。

### 5.2 代码示例：典型的生产者-消费者模型 (模拟相机与编码器)

```c++
#include <iostream>
#include <pthread.h>
#include <unistd.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cond = PTHREAD_COND_INITIALIZER;

bool has_new_frame = false; // 共享标志位

// 消费者：硬件编码器线程
void* encoder_thread(void* arg) {
    std::cout << "[Encoder] 启动，等待新帧..." << std::endl;
    
    pthread_mutex_lock(&lock);
    
    // 为什么要用 while 而不是 if？
    // 为了防止“虚假唤醒(spurious wakeup)”，这是 Linux 内核调度的坑，标准写法必须是 while
    while (!has_new_frame) { 
        // 1. 解锁 lock
        // 2. 线程休眠，等待 cond 信号
        // 3. 收到信号唤醒后，重新自动加锁 lock
        pthread_cond_wait(&cond, &lock); 
    }
    
    std::cout << "[Encoder] 收到新帧，开始编码推流！" << std::endl;
    has_new_frame = false; // 消费掉这帧数据
    
    pthread_mutex_unlock(&lock);
    return NULL;
}

// 生产者：相机传感器线程
void* camera_thread(void* arg) {
    sleep(2); // 模拟摄像头曝光和读取数据的耗时
    
    pthread_mutex_lock(&lock);
    has_new_frame = true; // 准备好数据了
    std::cout << "[Camera] 抓取到新的一帧图像！" << std::endl;
    pthread_mutex_unlock(&lock);
    
    // 发送信号，唤醒正在 cond 上等待的线程
    pthread_cond_signal(&cond); 
    
    return NULL;
}

int main() {
    pthread_t enc_tid, cam_tid;

    pthread_create(&enc_tid, NULL, encoder_thread, NULL);
    pthread_create(&cam_tid, NULL, camera_thread, NULL);

    pthread_join(enc_tid, NULL);
    pthread_join(cam_tid, NULL);

    return 0;
}
```

## 6. 总结

1. **`pthread` 是底层基石**：虽然 C++11 之后我们更推荐用 `std::thread`、`std::mutex` 和 `std::condition_variable`，但由于很多优秀的开源底层库（如 FFmpeg、各类硬件 SDK）是用纯 C 写的，看懂 `pthread` 是必修课。
2. **C++ 的对应关系**：
   - `pthread_create` $\rightarrow$ `std::thread`
   - `pthread_mutex_t` $\rightarrow$ `std::mutex`
   - `pthread_cond_t` $\rightarrow$ `std::condition_variable`
   - C++ 的封装大量使用了 RAII 机制（比如 `std::lock_guard`），可以自动帮你解锁，防止死锁（忘记写 unlock 的情况）。
